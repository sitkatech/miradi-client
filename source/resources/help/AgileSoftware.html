Agile Software Development

A.  The User's Perspective

When the members of the Conservation Measures Partnership first started discussing this program, not knowing much about software development, we bought a copy of the Dummies Guide to Software Project Management. This guide basically said that we had to work through the following steps:

1. Gather all requirements

  2. Design the entire system

    3. Write the code for the application

      4. Test and debug the entire application

        5. Deploy the application which will now work

Even at the time, this "waterfall" method of programming - so named because once you complete one step, there is no going back to it - seemed like a difficult thing to do. We weren't sure exactly what we wanted the program to do, so it seemed hard to design the entire system.

When we met with the Benetech team, they told us about a relatively new programming method called Agile Development, typified by a specific process know as Extreme Programming (XP). As outlined in the following diagram, Agile Development involves going through an iterative process of defining user stories (things that people want the software to do), writing modules of code in weekly iterations, testing these iterations, and then using the results to refine both the code and your list of stories.

Light bulbs started going off - here in the world of software was the same adaptive management approach that we have been advocating in conservation!! And interestingly, it has evolved for the same reason - a need to make decisions in the face of complexity and uncertainty. There is no way two years ago that we could have predicted that we would need many of the features that are in the current release of the software - we could only arrive at them through a systematic, iterative process.

Just as adaptive management is not a license to do "whatever you want," but instead must carefully state your assumptions and then deliberately test them, so too Agile Programming requires a disciplined approach. We the users can suggest any stories we want. But the engineers will then log them into our tracking system and provide estimates of how much programming time will be required to complete each step. We then meet with the engineers before each weekly iteration to collectively decide what stories we will do that week, trying to get the most "user value" for that week's worth of engineering time. We users also then have to make sure that we test and provide feedback on the results of that weeks' iteration.

We're not sure that this agile process would work with a much larger project requiring dozens of programmers versus the handful that we worked with. And it is obviously clearly dependent on having software engineers who are deeply skilled, enthusiastically willing to keep changing things, able to dive in and rapidly get up to speed on new subject material, ask intelligent questions, be problem solvers, and make substantial contributions to design and function elements that enrich submitted stories as well as produce new stories. But with those caveats, the agile process seems to work really well. 

So why are we providing you with all this navel gazing about our development process? There are two important messages we wish to convey.

1.  We Need Your Stories - Because we are using Agile Programming, we collectively have the power to change whatever we want in this software program. If you the user find something minor that bothers you (when you click on a factor, the wrong thing is selected) or have a grand idea of how to improve the software (adding an entire module that does....), you should know that we can make it happen. Drop us a note at feedback@miradi.org. We won't guarantee that we will satisfy all requests. But we will certainly consider it - and at least get back to you to discuss it with you.

2.  Agile Programming / Adaptive Management Works - Although we are preaching the gospel of Adaptive Management in conservation, we (as of yet) still only have a limited number of stories showing that this approach adds value. Our experiences with Agile Programming have left us convinced that in the face of uncertainty and complexity, a flexible and yet deliberate iterative approach is the only way to go. We can't imagine undertaking another software program any other way - and it give us faith that we are on the right track in conservation too.

B.  The Engineer's Perspective

Agile software development has been around for decades, but was only recognized and named a few years ago. In 2001, practitioners of so-called "lightweight" methodologies got together and created the Agile Manifesto. Since then, Agile development has continued to gain popularity and credibility, especially for projects with fewer than ten developers.

Shifting to Agile Programming from a more traditional methodology can be very challenging. Users take on more responsibility for prioritizing, and are expected to remain actively involved throughout the entire development cycle. Programmers have to let go of the idea that everything can be planned up front, and be willing to change direction as required. In order for that to be possible, the source code must always be kept clean and simple, and most agile projects use extensive automated tests to ensure high quality.

Extreme Programming (XP) is one specific form of Agile development. It is so famous that some people think that all Agile development must be "extreme". XP can work well, but it is very difficult to implement successfully, and is not appropriate for all teams or projects. For Miradi, we used a subset of XP that is also a variant of the Crystal Clear methodology.

For us, the benefits of Agile Programming include:

  • Early delivery of working software
  • Responsiveness to changing requirements
  • Low defect counts
  • Efficient use of funds
  • Avoiding end-of-cycle "death marches"
  • Continuous learning and improvement
  • More fun!
